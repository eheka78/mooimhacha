# 태스크 기한 연장 요청 + 팀장 승인 — 설계

작성일: 2026-06-14

## 목적

기한이 지난 태스크(액션 아이템)에 대해 **담당자가 연장을 요청**하고, **팀장이 수락/거절**한다. 수락하면 태스크의 기한(`due_date`)이 요청한 날짜로 바뀐다.

## 배경 (기존 구조)

- `ActionItem`: `due_date`(기한), `status`(todo/in_progress/done/cancelled), `assignee_id`, `team_id`. `action-items` 모듈에서 CRUD.
- 팀장 권한 체크는 `teamsService.requireLeader(userId, teamId)`가 이미 있다.
- 기여도(마감 준수)는 외부 엔진([cc-team-8/Contribution](https://github.com/cc-team-8/Contribution))이 `due_date` 기준으로 계산한다 — 우리 산정 로직은 수정하지 않는다.
- 요청→승인 흐름은 출결의 사유 결석 동의와 유사하나, 승인자가 과반이 아니라 **팀장**이다.

## 설계 결정 (확정)

1. **새 기한** = 요청자가 희망 기한을 제시. 팀장은 그 기한으로 수락하거나 거절.
2. **요청 주체** = 해당 태스크의 담당자(`assignee_id`) 본인만.
3. **팀장 처리** = 수락·거절 둘 다. 상태 `pending | approved | rejected`.
4. **기여도** = 수락으로 `due_date`가 바뀌면 외부 엔진이 자동 반영. 산정 로직 미수정.
5. **UI** = 태스크 카드 인라인 — 요청 버튼·대기 표시·팀장 수락/거절.

## 데이터 모델 (신규)

```
TaskExtensionRequest
  id              PK
  action_item_id  FK
  requester_id    FK (담당자 본인)
  requested_due_date  datetime (희망 새 기한)
  reason          text (요청 사유)
  status          enum: pending | approved | rejected
  created_at
```

같은 태스크에 `pending` 요청은 최대 1건(중복 차단). 거절/수락 후 재요청 가능 — 이력은 행으로 남는다.

## 플로우

1. **요청** — 담당자가 기한 지난·미완료 태스크에 희망 기한 + 사유 입력 → `pending` 생성.
   - 검증: 요청자 == 담당자, `status ∈ {todo, in_progress}`, `due_date != null && due_date < 현재`, 희망 기한 > 현재 `due_date`, 그 태스크에 `pending` 없음.
2. **수락(팀장)** — `ActionItem.due_date` = `requested_due_date`, 요청 `approved`.
3. **거절(팀장)** — 요청 `rejected`. 태스크는 그대로(기한 지난 상태 유지), 담당자는 재요청 가능.

### 상태 전이
```
(없음) --담당자 요청--> pending --팀장 수락--> approved (due_date 변경)
                          └------팀장 거절--> rejected (재요청 가능)
```

## API (신규 모듈 `task-extensions`)

| 메서드 | 경로 | 권한 |
|--------|------|------|
| POST | `/api/action-items/:id/extension` | 담당자 본인 (body: `requested_due_date`, `reason`) |
| GET | `/api/teams/:id/extensions?status=pending` | 팀 멤버 (팀장 처리 목록 + 조회) |
| POST | `/api/extensions/:id/approve` | 팀장만 |
| POST | `/api/extensions/:id/reject` | 팀장만 |

`GET /extensions` 응답(건별): `id`, `action_item_id`, `requester_id`, `requester_name`, `task_description`, `current_due_date`, `requested_due_date`, `reason`, `status`, `created_at`.

### 권한·검증
- 요청: 요청자 == 태스크 담당자, 위 요청 조건 충족.
- 수락/거절: `teamsService.requireLeader(userId, team_id)`. `pending` 상태만 처리(아니면 현재 상태 반환, 멱등).

## UI (태스크 카드 인라인)

태스크 카드에서:
- **기한 지남 + 미완료 + 본인 담당** → "연장 요청" 버튼 → 모달(희망 기한 + 사유)
- **요청 `pending`** → "연장 대기 중 (희망: M/D)" 배지
- **요청 `rejected`** → "연장 거절됨" 표시 + 재요청 가능
- **팀장 + 그 태스크에 `pending` 요청** → 카드에 "수락 / 거절" 버튼 + 희망 기한·사유 노출

## 범위 밖 (YAGNI)

- 연장 횟수 제한 없음.
- 알림(요청/수락/거절) 연동은 후속.
- 팀장이 수락 시 기한 수정 불가 — 요청 기한 그대로(수정형은 후속).
- 담당자 없는(`assignee_id = null`) 태스크는 연장 요청 대상 아님.

## 테스트 관점

- 요청 조건: 기한 안 지난/완료/타인 담당/담당자 없음 → 거부. 정상 → pending.
- 희망 기한 ≤ 현재 due_date → 거부.
- pending 중복 → 거부.
- 수락 → due_date 변경 + approved. 거절 → rejected, due_date 불변.
- 권한: 비담당자 요청 거부, 비팀장 수락/거절 거부.
